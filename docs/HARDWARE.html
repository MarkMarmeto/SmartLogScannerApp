Cameras vs. presentation scanner — which UX wins? Cameras give a live preview the student can self-correct against ("I can see my phone on the screen, move it closer"). For students unfamiliar with QR kiosks (especially younger grades), the camera preview is more forgiving. For high-volume known-cohort gates, presentation scanners are 2–3× faster per scan.
⚠️
Critical Caveat — Multiple Keyboard-Wedge Devices
Do not run 2+ HID keyboard-wedge scanners (handheld or presentation) simultaneously without code changes. Every HID barcode device — handheld, fixed presentation, or pedestal — injects keystrokes into the focused text field exactly like a real keyboard. If two scans arrive within ~150 ms of each other, the characters interleave in the input buffer, producing a corrupted payload that fails HMAC validation. The form factor (handheld vs. fixed) does not change this; the USB interface class does.

Why the app currently tolerates only one

The Scanner mode (PreferencesService.ScannerMode = "USB") listens for keystrokes on a single focused entry control. Windows does not tell the app which physical device produced each keystroke — both scanners are seen as anonymous keyboard input.

Options if you ever need multiple presentation scanners

  • Recommended (no code change): Use at most one keyboard-wedge scanner per host PC. Add more cameras instead — cameras are independent decode pipelines that don't share the keyboard buffer.
  • Code change required: Switch to the Windows Raw Input API (WM_INPUT) to attribute each keystroke to a specific HID device handle. This lets each scanner feed its own decode queue. Estimated effort: 2–3 days plus testing on real hardware.
  • Hardware alternative: Use scanners in USB Serial (COM port) mode — Honeywell Genesis 7580g, Zebra DS9308, etc., can be configured to expose each device as its own COM port, completely sidestepping the keyboard buffer. Most Eyoyo models do not reliably support COM mode.
Implication for the hands-free plan If you adopt the supplemental fixed presentation scanner, keep it to one unit. Scale by adding cameras (independent USB Video pipelines), not more wedge scanners. If multiple kiosk lanes are required, either deploy separate host PCs (one scanner each) or commit to the Raw Input rewrite.

Webcam Selection

How many cameras do you actually need?

QR decode speed is bounded by three things, in this order of usual impact:

  1. Optics & autofocus — a sharp 720p frame decodes faster than a blurry 1080p frame.
  2. Lighting — even illumination beats camera quality; see § Lighting.
  3. Number of decode workers — diminishing returns past 2 lanes per gate.
Camera countThroughput (typical)Verdict
1 camera~25–35 students/minOK for small schools (< 300 students); single point of failure.
2 cameras~50–70 students/minRecommended Best throughput-per-dollar; redundancy if one fails.
3 cameras~70–90 students/minUseful only if students physically queue in 3 parallel lanes. Otherwise the extra camera is idle.
4+ camerasmarginal gainHits CPU and USB-bandwidth ceilings on mid-range hardware; AdaptiveDecodeThrottle will start downscaling frame rates.
Counter-intuitive: three mediocre webcams (slow autofocus, fixed-focus lenses, noisy sensors) can deliver worse aggregate throughput than two good ones — wasted decode CPU on unfocused frames hurts every lane.

Key technical requirements

  • UVC-compliant (USB Video Class) — works without vendor drivers; required by Windows MediaFoundation pipeline.
  • MJPEG output supported — uses ~10× less USB bandwidth than uncompressed YUY2 at the same resolution. Critical when running 3 cameras on shared controllers.
  • Autofocus or fixed focus calibrated for ~30–60 cm — that is the realistic distance from camera to a phone screen showing a QR.
  • Stable USB device path — the multi-camera service distinguishes identical-model webcams by device path, not friendly name.
Resolution sweet spot 720p is sufficient for QR codes printed on student ID cards or shown on phone screens at typical scan distances. Configure cameras to 720p in SetupPage if 1080p shows no improvement in decode latency — you save ~50% of USB bandwidth and CPU per camera.

Fixed Presentation Scanner (Optional)
Hands-free constraint A presentation scanner is a fixed, hands-free unit mounted on the gate counter. The student presents the QR to it themselves — no operator. Think supermarket self-checkout barcode reader. This is only suggested as a supplement to cameras, not a replacement.

When to add one (and when not to)

  • Add one if: the student demographic is older / familiar with kiosks, queues are long, or webcam-only trials show students struggling to position phones in the camera's focus zone.
  • Skip it if: students are young (kindergarten/elementary), the gate is well-lit, or you want a single uniform UX across all lanes. Two cameras alone handle most schools comfortably.
  • Never deploy 2+: see the Critical Caveat — multiple wedge devices on one PC corrupt scans.

Mounting a presentation scanner

  • Mount at chest height (~110–130 cm) on a fixed stand or counter, scan window angled ~30° upward toward the approaching student.
  • Leave clear floor space ~40–60 cm in front so students naturally stop within scan range.
  • Add a printed sign with a phone-icon arrow showing where to present the QR — students take 2–3× longer the first time without visual instruction.
  • Run the USB cable internally through the stand to avoid trip hazards and tampering.
USB receiver placement (if wireless) Most desktop presentation scanners are wired (recommended — no battery to fail). If you do choose a wireless variant, plug its USB receiver into a front-panel USB 2.0 port, not USB 3.0. USB 3.0 controllers emit RF interference in the 2.4 GHz band that can shorten wireless range by 3–5 m.

Why a mini-PC over a laptop?

  • Thermals: The decode pipeline + UI is sustained CPU load. Laptops thermally throttle within 10–15 min; mini-PCs run flat-out indefinitely.
  • USB ports: Most laptops expose 2 USB-A ports — not enough for 3 cameras + 1 scanner + keyboard/mouse without a hub.
  • Cost: A new i5 mini-PC is typically cheaper than a comparable-CPU laptop and lacks a screen/battery you don't need.
  • Mounting: Mini-PCs have VESA brackets — mount behind the gate display and run cables cleanly.

Why not a Raspberry Pi?

Tempting for cost, but:

  • .NET 8 MAUI Windows is the supported deployment target. A Pi would require a Linux port (significant rewrite).
  • Single shared USB 2.0 controller — 3 cameras will saturate it.
  • ARM CPU is decode-throttled even with 1 camera at 1080p.

USB Topology & Bandwidth

The hidden bottleneck

Camera count is rarely the limit. The limit is which physical USB controller each camera lands on. All ports on one controller share a single bandwidth pool.

USB standardTheoreticalPractical sustained720p MJPEG @30fps consumes
USB 2.0480 Mbps~280–320 Mbps~25–40 Mbps per camera
USB 3.0 (3.2 Gen 1)5 Gbps~3.2 Gbps~25–40 Mbps per camera
USB 3.0 with YUY2 (uncompressed)5 Gbps~3.2 Gbps~470 Mbps per camera (!)
YUY2 trap A single 1080p YUY2 webcam consumes ~950 Mbps. Three of them on one USB 2.0 controller will silently drop frames or fail to enumerate. Always verify cameras negotiate MJPEG on startup (logs report the chosen subtype).

Recommended topology for 3 cameras + 1 scanner

  • Camera 1 → motherboard rear USB 3.0 port (controller A)
  • Camera 2 → motherboard rear USB 3.0 port (controller B, if available)
  • Camera 3 → motherboard front USB 3.0 port (often a third controller via header)
  • Eyoyo wireless dongle → front USB 2.0 port (avoids 2.4 GHz interference)
  • Keyboard / mouse → any remaining USB 2.0 port

Check controllers in Device Manager → View → "Devices by connection". Each USB Host Controller is a separate bandwidth pool.

Cabling

  • Use the cables that ship with the camera; do not extend with cheap passive 5 m cables — they cause enumeration drops.
  • If the camera must be > 3 m from the PC, use an active USB 3.0 extension (with built-in repeater) — Cable Matters and StarTech make reliable ones.
  • Avoid daisy-chaining hubs. One powered hub between PC and devices is fine; two hubs in series often causes power negotiation failures.

Lighting, Mounting & Environment

Lighting matters more than camera spec

The cheapest reliable improvement to decode rate is even, diffuse light at the scan zone. Direct sunlight, harsh fluorescent flicker, and dark shadows all hurt decode rate more than a 50% camera resolution drop would.

  • Aim for 300–500 lux at the QR plane (a phone exposure-meter app can verify).
  • Avoid backlighting — never put the camera facing a window or open doorway during morning hours.
  • Add a small LED strip above the scan zone if the gate is shaded; the Razer Kiyo has one built-in.
  • Anti-glare matters: students presenting glossy phone screens at an angle into a low sun will fail to scan even on a perfect camera.

Mounting geometry

  • Camera at chest-to-shoulder height (~120–140 cm), tilted slightly downward.
  • QR presentation distance: 30–50 cm — verify autofocus locks at this range during calibration.
  • For 3-camera lane setup, space cameras ~80 cm apart so each owns a clear "lane" with no overlap.
  • Mark a floor sticker / footprint where students should stand — consistent geometry > clever software.

Display visibility

The student-feedback card must be readable at 2–3 m. The current 1080p layout is sized for that distance. If the host PC drives an external monitor, choose at least a 21" screen at 1080p facing the student queue.

🌐
Network Connectivity
  • Wired Gigabit Ethernet preferred. The app's BackgroundSyncService polls health every 15 s; Wi-Fi micro-disconnects (common in school networks) cause the offline queue to fill briefly even when nothing is wrong.
  • If Wi-Fi is mandatory, use 5 GHz SSID — 2.4 GHz collides with the wireless scanner and other school devices.
  • Whitelist the SmartLog backend hostname/IP in any school proxy. The Polly retry policy (3× exponential + circuit breaker at 5 failures / 30 s) handles transient blips, not sustained proxy outages.
  • Backup connectivity: a USB-tethered phone or 4G LTE dongle is a worthwhile insurance policy for high-traffic gate hours. The offline queue covers brief outages but a 30-minute outage at peak fills the queue.

When to add a second gate (separate PC) instead of more cameras

  • Required throughput > 60 students/min — physical queueing becomes the bottleneck, not decode rate.
  • One gate would require 4+ cameras — better to split traffic across two gates with 2 cameras each (also gives redundancy if one PC fails).
  • School has multiple physical entrances — one host PC per entrance scales linearly; the backend already deduplicates across devices via the tiered time-window dedup (3 s / 60 s / 300 s).

Validating in production

After deployment, the Statistics footer on MainPage reports per-camera scan counts. If one camera consistently shows < 15% of the total over a week, it is mis-positioned (or its lane is unused) — relocate or remove rather than leaving it idle consuming USB bandwidth and CPU.
